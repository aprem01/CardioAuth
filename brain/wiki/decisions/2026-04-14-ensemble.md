# 2026-04-14 — Self-consistency ensemble

**Status:** active
**Driver:** Product-readiness push — reasoner output needs to feel reproducible for a clinical tool

## Problem

UnifiedReasoner was a single Claude call per case. Re-running the same case could yield different scores. For clinical use, physicians will notice and lose trust. We also had no way to tell confident decisions from noisy ones.

## Decision

Implement self-consistency ensemble (Wang et al. CoT):

- `REASONING_ENSEMBLE_N` env var (default 1 for dev, set to 3 in production).
- Each run uses temperature 0.3 for diversity.
- Majority vote per criterion; tie → `not_met` (conservative default).
- Per-criterion `agreement_score = count_agreeing / N`.
- Case-level `agreement_score = mean of per-criterion`.
- Criteria with agreement < 0.67 surface to cardiologist as `low_agreement` flag.

## Consequences

- API cost ×3 per case. Acceptable for clinical use; would re-evaluate if scale pushes cost.
- New endpoint `POST /api/validation/consistency` runs one case N times and reports variance — the "is this system reproducible?" artifact for compliance.
- Agreement score flows into the audit trail and API response.
